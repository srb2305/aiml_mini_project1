1. Problem Statement

1.1  The Real-World Pain Point

Enterprise teams — in legal, finance, healthcare, and research — sit on thousands of documents: contracts, regulatory filings, research papers, SOPs, and internal reports. They need to query this knowledge continuously to make decisions.

Current RAG pipelines answer simple factual queries but fail at relationship-heavy questions such as:

“Which vendors are linked to compliance risk flagged in Q3 reports?”

“How does policy A affect regulation B through clause C?”

“What research findings contradict our current clinical protocol?”

These fail because vector search retrieves similar text but has no understanding of how entities relate to each other. It treats every chunk as an island.

1.2  Three Hard Limitations of Standard RAG

2. Solution Overview

You will build a hybrid retrieval system that combines a vector database and a knowledge graph, runs a reranker over merged candidates, generates answers via an LLM, critiques its own output through a LangGraph self-correction loop, and logs automated evaluation scores to a live dashboard.

4. System Architecture

4.1  Ingestion Pipeline

When a document is uploaded, two parallel pipelines run simultaneously:

Vector Pipeline

Load document with LangChain loaders (UnstructuredPDFLoader, Docx2txtLoader, etc.)

Semantic chunking — split at meaning boundaries using embedding similarity

Embed each chunk with sentence-transformers

Store chunk + embedding + metadata (source, page, timestamp) in pgvector

Graph Pipeline

Run spaCy NER over each chunk to extract entities (persons, orgs, locations, concepts)

Run an LLM prompt to extract relationships: “Extract (entity1, relation, entity2) triples from this text”

Store entities as nodes, relationships as edges in Neo4j

Link each node back to the source chunk ID in pgvector for cross-referencing

4.2  Query & Retrieval Pipeline

When a user submits a query, three retrieval steps run before generation:

Step 1 — Parallel Dual Retrieval

Vector retrieval: embed query, top-K cosine similarity search in pgvector

Graph retrieval: extract query entities via NER, run Cypher traversal in Neo4j to find connected nodes up to 2 hops, fetch linked chunks

Merge and deduplicate both candidate sets

Step 2 — Reranking

Pass merged candidate set through CrossEncoder (ms-marco-MiniLM-L6-v2)

CrossEncoder scores each (query, candidate) pair independently — much more accurate than cosine similarity

Select top-N reranked chunks as final context

Step 3 — Generation with Self-Correction Loop (LangGraph)

Goal

Wrap retrieval and generation in a stateful LangGraph that critiques its own answers and re-retrieves if quality is below threshold.

Steps

Define LangGraph state schema: { query, context_chunks, answer, faithfulness_score, relevance_score, iteration_count }

Node 1 — retrieve: calls HybridRetriever, stores chunks in state

Node 2 — generate: formats prompt with context + query, calls LLM, stores answer in state

Node 3 — critique: calls judge LLM with prompt: “Given this context and this answer, rate faithfulness (0-1) and relevance (0-1). Return JSON.”

Node 4 — decision (conditional edge): if scores ≥ 0.7 → log and return. If scores < 0.7 and iteration < 2 → refine query and go back to Node 1

Node 5 — log: write full trace to LangSmith, write eval scores to MLflow

Set max_iterations = 2 to prevent infinite loops

Test: deliberately ask ambiguous questions and observe whether the loop refines and improves

Milestone

✅  You can observe in LangSmith that the system re-retrieves on low-confidence answers. Scores improve on iteration 2.
